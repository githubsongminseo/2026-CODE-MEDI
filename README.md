# CODE:MEDI CPX 프로토타입 — 실행 가이드 (v3, "나쁜 소식 전달" 전용 재설계)

교수님 피드백을 반영해서 **질병에 대한 전반적 문진이 아니라, 나쁜 소식 전달 + 감정 대응
의사소통에만 집중**하도록 전체를 다시 설계했습니다. 이전 버전(11개 케이스, F1-F8 체계)은
`backend/cases_archive_v1/`, `backend/checklist_reference_v1_archive.json`에 보관되어
있고, 지금 실행되는 건 새 케이스 체계(현재 10개)입니다.

## 이전 버전과 달라진 점

1. **학습자가 먼저 대화를 시작**합니다 ("안녕하세요. 담당 의료진 OOO입니다..."). 환자가
   먼저 말을 거는 일이 없습니다.
2. **케이스는 "객관적으로 충격적인 진단"만으로 제한**됩니다 (현재 10개 — 아래 목록 참고).
   초기 6개는 새 진단 전달, 추가된 4개(B07~B10)는 재발/감염병/조기진단/태아진단 등으로
   범위를 넓혔습니다.
3. **환자의 초기 감정 반응이 케이스가 아니라 매 세션마다 랜덤으로 배정**됩니다 (부정/분노/
   협상/우울 중 1개, 케이스와 완전히 독립적인 추첨). 대화 중 이 4개 사이를 전환하는 일은
   없습니다 — 같은 감정 안에서 강도만 오르내리다가, 학습자가 잘 대응하면 "수용"으로
   수렴합니다. 강도는 평소 CPX보다 훨씬 크게 설정됩니다.
4. **체크리스트가 훨씬 단순해졌습니다** — 케이스별 후보 체크리스트(C1-C9)가 사라지고,
   모든 케이스에 동일하게 적용되는 11개 핵심 항목(E1-1~E3-1) + 감정반응별 2개 항목
   (부정/분노/협상/우울/수용 각 2개)만 사용합니다.
5. **환자는 차트 밖의 깊은 의학 질문(약물명, 구체적 시술 등)을 하지 않습니다.** 가벼운
   걱정성 질문만 합니다.
6. 난이도(하/중/상)는 케이스/감정 선택에 영향을 주지 않습니다 — **(1) 감정 단서의
   명시성(지시문 빈도), (2) 수용으로의 설득 저항도** 두 가지를 조절합니다. 감정 강도
   자체는 모든 난이도에서 항상 높게 유지됩니다.

## 구조
```
cpx-prototype/
  backend/
    main.py                       # FastAPI 서버
    cases/                        # 케이스 10개 (B01~B10), 전부 ready_for_play=true
    cases_archive_v1/             # 이전 11개 케이스 (보관용, 로드되지 않음)
    checklist_reference.json      # 체크리스트: core(11) + emotion(5×2) + CF(8) + PPI(6)
    checklist_reference_v1_archive.json  # 이전 체크리스트 (보관용)
    reports/                      # 평가 레포트 DB
    generate_synthetic_reports.py # 합성 대화+레포트 자동 생성

    requirements.txt
    .env.example
  frontend/
    index.html                    # 난이도 선택 → 채팅(학습자 선시작) → 리포트
```

### 케이스 페르소나 구조 (핵심)
각 케이스의 `patient_persona`는 다음을 갖습니다:
- `initial_emotion`: `{category, intensity_note, example_lines, no_switch_rule}` — 고정된
  단일 초기 반응. `no_switch_rule`이 "다른 3개 범주로 전환 금지"를 프롬프트에 명시함.
- `convergence_to_acceptance`: `{trigger_condition, example_lines}` — 수용으로 수렴하는
  조건과 그때의 예시 발화. 별도의 "전환 가능한 감정"이 아니라 엔드포인트 상태.
- `spikes_eliciting_hooks`: 환자의 행동이 학습자의 SPIKES 단계를 자연스럽게 유도하도록
  하는 행동 지침 (예: 침묵하기, 동석자 여부를 스스로 언급하기 등).
- `shared_behavior_rules`: 모든 케이스 공통 — 차트 밖 깊은 질문 금지, 질문 억지로 만들지
  않기, 먼저 대화 시작하지 않기.

### 체크리스트 코드
- **E1-1~E1-4** (첫 멘트): 인사/역할소개, 신원확인, 동석자확인, 목적설명+준비확인
- **E2-1~E2-6** (중간): 감정확인질문, 쉬운용어, 공감/지지, 치료계획설명, 이해확인, 침착한태도
- **E3-1** (마무리): 추가질문 확인 후 마무리
- **감정반응**: 부정(D1,D2) / 분노(A1,A2) / 협상(B1,B2) / 우울(DEP1,DEP2) / 수용(AC1,AC2)
  — 대화에서 실제로 등장한 감정만 평가, 등장 안 한 건 결과에서 생략
- **CF1~CF8**: Critical Fail (이전과 동일, 변경 없음)

## 실행 방법

### 1. 앱 띄우기
```bash
cd backend
pip install -r requirements.txt
cp .env.example .env   # OPENAI_API_KEY 입력
uvicorn main:app --reload --port 8000
```
`http://localhost:8000/` 접속 시 프론트엔드가 열립니다. `http://localhost:8000/api/health` 접속 시
`{"status":"ok","total_cases_loaded":10,"playable_cases":[...10개...]}` 확인.

## 사용 흐름
1. 난이도(하/중/상) 선택 — 감정 단서 명시성만 조절.
2. 채팅창이 빈 상태로 뜸. **학습자가 먼저** "안녕하세요. 담당 의료진 OOO입니다..." 로 시작.
3. "📋 환자 차트 보기"로 검사결과/치료계획 확인 가능.
4. 대화를 진행하다 환자가 "네, 알겠습니다" 류로 마무리 의사를 보이거나, 학습자가 직접
   마무리하면 "문진 종료하고 평가받기" 클릭.
5. 체크리스트(11항목) + Critical Fail + 감정반응 대응(등장한 감정만) → PPI 6항목 + 서술형
   피드백 → 약점분석(면담구조/정보탐색/정보전달/감정대응) → 다른 감정의 케이스 추천.
6. "OO 케이스로 재도전" 버튼으로 추천받은 그 케이스로 바로 이어서 연습 가능.

## 알아두어야 할 제약
- 세션은 메모리에만 저장(서버 재시작 시 소실).
- 평가는 대화 종료 후 한 번에 진행(사후 일괄).
- "환자가 4단계(나쁜소식 전달→감정 격화→수렴→간단한 후속질문→SPIKES 유도)를 거쳐 자연스럽게
  마무리를 제안한다"는 엔드포인트 로직은 **프롬프트 지침으로만 구현**되어 있습니다 — 실제로
  매 케이스에서 일관되게 그렇게 동작하는지는 검증이 더 필요합니다. 특히 "차트 밖 질문을 하지
  않는다"는 제약이 실제 대화에서 얼마나 잘 지켜지는지 직접 테스트가 필요합니다.
- 10개 케이스 전부 본질적으로 무겁고 격한 감정이라, 난이도 "하"에서도 감정 강도 자체는
  여전히 강합니다 (의도된 설계입니다 — "하"는 강도가 아니라 단서 명시성만 낮춥니다).
- OpenAI API 모델명/요청 형식이 바뀌면 `main.py`의 `call_openai()`를 최신 문서 기준으로
  맞춰야 합니다.

## 실시간 transcript 저장

세션이 시작되면 `backend/transcripts/{session_id}.json` 파일이 만들어지고, 학습자 발화와
환자 응답이 추가될 때마다 같은 파일이 갱신됩니다. 평가를 누르지 않아도 대화 기록은 이
파일에 남습니다.

- `GET /api/transcripts` — 실시간 저장된 transcript 목록(메타데이터)
- `GET /api/transcripts/{session_id}` — 해당 세션 transcript 전체

## 레포트 저장 (DB) 및 합성 데이터 생성

평가가 끝날 때마다 **대화 스크립트 전체 + 체크리스트 + 감정반응 + PPI + 약점분석**을 한
묶음으로 `backend/reports/{report_id}.json`에 저장합니다. 의대생 측에서 만들 "실제
연극배우용 PPI 판단 프롬프트"의 입력 데이터로 이 파일을 그대로 쓸 수 있게, transcript와
평가 결과를 분리하지 않고 한 파일에 묶어둡니다.

- `GET /api/reports` — 저장된 레포트 목록(메타데이터)
- `GET /api/reports/{report_id}` — 레포트 전체 (transcript 포함)
- 학습자가 직접 플레이한 건 `source: "live"`, 아래 합성 생성기로 만든 건
  `source: "synthetic"`으로 구분됩니다.

### 합성 데이터 생성기 (사람이 매번 채팅 안 쳐도 됨)
`backend/generate_synthetic_reports.py`가 "학습자 역할 LLM"과 "환자 역할 LLM"을 자동으로
대화시켜서 레포트를 대량 생성합니다. 학습자 역할은 우수/보통/미흡 3단계 실력으로 시뮬레이션
되어, 다양한 품질의 샘플을 한 번에 확보할 수 있습니다.

```bash
cd backend
python3 generate_synthetic_reports.py --count 5
# 특정 케이스/실력/난이도 고정도 가능
python3 generate_synthetic_reports.py --count 1 --case-id B01-pancreatic --skill 미흡
```

대화 1턴당 학습자+환자 LLM 호출이 각각 1번씩, 평가 시 추가로 2번 호출되므로 비용이 발생함을
참고하세요 (대화 10턴 기준 약 22회 호출).

### 이미 가진 대화를 평가만 다시 돌리고 싶을 때
`backend/evaluate_transcript.py` — 이미 끝난 대화 transcript(JSON)를 입력받아 채점만
다시 돌리고 레포트로 저장합니다. 스크린샷 등에서 옮긴 과거 대화를 정식으로 평가해서
DB에 넣고 싶을 때 씁니다.

```bash
cd backend
python3 evaluate_transcript.py --input reports/28af8607-105b-4481-ae82-c930776590c1_input_transcript.json --difficulty 중
```

### 모델 2단계 분리
`.env`에서 두 모델을 따로 설정합니다:
- `OPENAI_MODEL_CHAT` — 환자 역할 대화 생성용. 턴마다 호출되니 가볍고 빠른 모델 권장 (기본값 `gpt-4o-mini`).
- `OPENAI_MODEL_EVAL` — 체크리스트/PPI 평가용. 더 정교한 판단이 필요해서 무거운 모델 권장 (기본값 `gpt-4o`).
